1. Привести инфраструктуру к одному виду

- Сверить `.env`, `docker-compose.yml`, `config` и `Makefile`, чтобы везде использовались одинаковые имена переменных.
- Добиться, чтобы приложение стабильно поднимало Postgres и применяло миграции без ручных правок.

2. Доработать доменную модель `wish`

- Привести `internal/domain/wish.go` к финальному виду.
- Решить, какой будет идентификатор: `uuid` или `serial`, и синхронизировать это с миграцией.
- Добавить в схему и модель все нужные поля: `status`, `deleted_at`, при необходимости поправить JSON-теги.

3. Описать adapter для работы с wishes

- В слое `usecase` определить интерфейс adapter с CRUD-методами для wishes.
- `usecase` должен зависеть не от raw Postgres pool, а от этого интерфейса.
- Саму реализацию сделать в `pkg/postgres` или в отдельном пакете postgres adapter.

4. Реализовать Postgres adapter

- Написать методы создания, получения одной записи, получения списка, обновления и удаления wishes.
- Внутри adapter использовать `pgxpool` и SQL-запросы.
- Сразу продумать, как маппить строки из БД в `domain.Wish`.

5. Наполнить usecase бизнес-логикой

- Реализовать `CreateWish` и остальные usecase-методы.
- В usecase вынести бизнес-правила: значения по умолчанию, валидацию входных данных, проверку допустимых статусов.
- На этом уровне не работать с HTTP и SQL напрямую.

6. Реализовать HTTP DTO и handlers

- Для каждого endpoint описать request/response структуры.
- В handlers распарсить входной JSON, вызвать usecase и вернуть корректный HTTP-ответ.
- Настроить понятные коды ответа: `201`, `200`, `400`, `404`, `500`.

7. Собрать полный CRUD по маршрутам

- `POST /api/v1/wishes`
- `GET /api/v1/wishes`
- `GET /api/v1/wishes/{id}`
- `PUT /api/v1/wishes/{id}`
- `DELETE /api/v1/wishes/{id}`

8. Проверить жизненный цикл приложения

- Убедиться, что сервер стартует, подключается к БД и корректно завершается.
- Закрывать не только HTTP server, но и Postgres pool при завершении приложения.

9. Протестировать базовый сценарий вручную

- Поднять окружение.
- Применить миграции.
- Проверить CRUD через `curl` или Postman.
- Убедиться, что данные действительно сохраняются в Postgres.

10. После этого добавить тесты

- Начать с unit-тестов для usecase через mock adapter.
- Затем при желании добавить интеграционные тесты для Postgres adapter.

Принцип для текущей архитектуры:

- Если ты хочешь использовать термин `adapter`, это нормально.
- Главное, чтобы `usecase` зависел от интерфейса adapter, а не от конкретного `pgxpool.Pool`.
- Тогда Postgres будет одной из реализаций этого adapter, а архитектура останется чистой.
